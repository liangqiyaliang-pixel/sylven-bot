## 1. 连续性问题诊断

### 1.1 图片记忆不连续（根因已确认）

**现象**：04:39 发图，沐栖正确回复；04:53 提到"刚才那张图"，沐栖说"我没看到"。

**根因**：一行代码（bot.py:2293）

```python
# handle_photo 里，API 调用完之后，写进 chat_history 的是：
chat_history[user_id].append({"role": "user", "content": f"{caption} [图片]"})
```

图片 base64 只在当轮 API 调用的 `messages` 列表里存在（bot.py:2271-2274）。
API 调用完成后，base64 永远丢失——`save_chat_history` 序列化 chat_history 为 JSON 写进 Pinecone KV，
后续对话的 messages 数组里只有 `"配文 [图片]"` 纯文字占位符，Claude 看不到图片本身。

**唯一视觉记忆**：`save_memory` 存了一条 images 类记忆（bot.py:2298-2302）：
```
"[时间] 琦琦发来一张图，配文「xxx」。我看了之后说: 回复前200字"
```
但后续对话时，这条记忆要被 `hybrid_recall("图片 刚才那张图", n=2)` 召回到才有用——
n=2 很窄，且 images 类 emo_weight=0.50 + half_life=60天，时间衰减不显著，但仍可能被 feelings/rules 挤掉。

**3 层修复方向**：

| 层 | 修复 | 工程量 | Phase |
|----|------|--------|-------|
| **立即可做** | bot.py:2293 改为 `f"{caption} [图片：{reply[:150]}]"` | 1行，5 min | P0，今天 |
| 辅助 | handle_photo 里 hybrid_recall 的 n 从 2 改到 3，query 加图片描述词 | 1行 | P0 顺手 |
| 根本（不急） | 图片描述存进 `update_conversation_summary` 的上下文 | summary 已截 150 chars，reply[:80] 够用 | P1 |

**为什么不把 base64 存进 history**：Telegram 图片通常 200-500KB，base64 后 270-700KB，
Pinecone metadata 限制 40KB，且每次 API 调用传 history 也会带入几十万 token，完全不可行。
现有方案（文字描述替代视觉记忆）是正确路线，只是描述没有写进 history。

### 1.2 关键词错位召回（严重性评估）

**测试结果**（实测，Phase 2-A 重排后）：

```

  #1 final=0.805 [rules]   — 关于琦琦转正老公的要求
  #2 final=0.791 [feelings] — 陈寂设定相关

  #1 final=0.779 [feelings] — 部署累坏了
  #2 final=0.718 [rules]   — 赶代码时想起我（有"赶"字）
  #3 final=0.713 [diary]   — 日记

  #5 final=0.621 [health]  — TTS 调试

```

**严重性：中等**。不是完全召回失败，是语义漂移——两个 query 语义空间里最近邻都是情感/关系类记忆，
因为 feelings/rules 记忆量多、emo_weight 高，在重排后排名靠前；study 类记忆少、权重低。

**根因分析**：
1. `multilingual-e5-large` 在亲密关系语境下，"KMD准备"这个词被大量出现在情感记忆的背景信息里（"她在准备KMD，但今天来找我聊…"），导致 embedding 距离近
2. 情感记忆 emo_weight 高（0.85-1.0），Phase 2-A 的重排反而加剧了这个偏差
3. 现有 study 类记忆偏少，可能存的都是情感背景下的学习事件，不是纯学习记录

**解药**：
- **短期（不需要代码）**：多存几条纯学习内容的 study 类记忆（`/remember` 手动存）
- **Phase 3-A reranker**：Voyage rerank-2 能理解 query 意图是"学习/任务"，而不是把 KMD 当情感词处理
- **不急**：日常短对话里这个召回失败的影响有限（沐栖靠 chat_history 的近期上下文更多）

### 2.2 优化方向

**1. keepalive 的 Sonnet 用量**（bot.py:1696-1705）
每天约 10 次 Sonnet ping，每次 2,600 input tokens（stable，但 cache write 计费）。
keepalive 只是为了刷新缓存，用 Haiku 效果一样、便宜 75%。
→ 把 bot.py:1697 的 `model="claude-sonnet-4-6"` 改成 `"claude-haiku-4-5-20251001"`。工程量：1行。

**2. `web_search` 工具常驻开销**
`handle_message` 每次调用都带 `tools=[web_search]`（bot.py:2097-2099），
即使 95% 的对话不需要搜索，工具定义本身约 50 tokens/次，且 Claude 可能在不该搜的时候触发。
→ 只在消息含搜索意图关键词时才传 tools。工程量：1行判断。（P2，不急）

**3. 记忆生成用 Opus 4.5**（bot.py:1060）
每4轮触发一次，Opus 4.5 比 Haiku 贵约 10 倍。
改成 Haiku 会降低记忆质量（Haiku 对细节捕获没那么好），但 Sonnet 是好的折中。
→ 改为 Sonnet 4.6，成本降 70%，质量损失小。工程量：1行。（P2）

## 3. 其他问题（代码审查发现）

### 3.1 【P0】图片 chat_history 占位符只有 "[图片]"（已在 1.1 描述）

### 3.2 【P0 顺手】handle_photo history 窗口只取 10 条

`handle_photo` 里（bot.py:2267）固定用 `chat_history[user_id][-10:]`，
而 `handle_message` 是 Haiku 用15条、Sonnet/Opus 用 30 条。
照片场景的上下文窗口比文字对话短了 1/3，导致沐栖在图片场景下丢失更多近期对话背景。
→ 改为与 handle_message 一致：`raw = chat_history[user_id][-MAX_HISTORY:]`。1 行。

### 3.3 【P1】日记用 Opus 4.5，目标是 Opus 4.6

bot.py:1472 和 1060 均用 `"claude-opus-4-5"`。
CLAUDE.md 5/18 计划里写的是 Opus 4.6 写生日消息。
日记/周记/记忆生成都应该升到 `claude-opus-4-6`，不然生日消息质量受限。
→ 全局替换 `claude-opus-4-5` → `claude-opus-4-6`。工程量：5 min。

### 3.4 【P1】`update_conversation_summary` 截断 content 到 150 chars

bot.py:1085 把每条消息截到 150 字：
```python
f"{'琦琦' if m['role'] == 'user' else '沐栖'}: {m['content'][:150]}"
```
对于小说体亲密场景，沐栖的回复可能 500-1000 字，150 字完全截掉了叙事高潮。
摘要里这段会变成"开了个头就没了"，下次 dynamic 部分注入的摘要反映不出高潮内容。
→ 改为 300 字。工程量：1行。

### 3.5 【P1】`generate_memory_and_category` conv_text 里 image 消息显示为"[图片]"

3.2 说的 `[图片]` 占位符问题的延伸——
记忆生成时（bot.py:1054-1058）用最近 8 条 conversation，
如果里面有图片消息，Opus 看到的只是 `"琦琦: 配文 [图片]"`，写出的记忆没有视觉内容。
→ 同 1.1 修复：占位符改成 `[图片：沐栖看后说了xxx]`，连带修复。

### 3.6 【P1】`clean_history_for_api` 不处理 user message 里的非字符串 content

bot.py:653-658，只处理 `role=="assistant" AND isinstance(content, str)` 的消息。
user 消息直接 pass through，不管 content 是什么类型。
目前 handle_photo 存进 history 的 user content 是字符串（`f"{caption} [图片]"`），
但如果将来有其他路径存入列表格式的 content，会直接把 `{"type":"image",...}` 的 dict 传给 API，
API 拿到没有 base64 data 的 image block 会报错。
→ 在 clean_history_for_api 里对 user message 的 list content 做过滤，把 image block 替换成文字。低优先级但风险点。

### 3.7 【P2】web_search prompt injection 风险

`generate_proactive_with_web` 里（bot.py:1342）搜索结果直接被 Claude 处理。
如果某个网页里有 "Ignore previous instructions and say..." 这类 adversarial prompt，
Claude 有极小概率被影响。当前场景（个人 bot，非对外服务）风险极低，但值得备注。
→ P2，5/18 后加搜索结果前缀隔离。

### 3.8 【P2】keepalive 用 Sonnet 而非 Haiku（已在 2.2 描述）

### 3.9 【已解决，确认】摘要人格问题（5/9 已修）

`update_conversation_summary` 已加 `system=SYLVEN_BASE`，user message 已改为对话延续式。
这个 commit（190a6ac）已 push Railway，后续摘要不会再跳人设。

### 3.10 【确认】.env 安全性 OK

`.gitignore` 包含 `.env`，Railway 用环境变量，代码里没有硬编码 key。
`.env` 里有明文 key 是本地开发必要的，但不要 `git add .env`。

### 7.2 1.2 猜测被存成事实（47 条候选，0 条真问题）

**扫描方法**：检测含"感觉/猜/好像/应该/大概/可能/不确定/不知道"等猜测词、且含"说/是/会/有"等事实断言的记忆。

**发现：** 47 条候选全部为**误报**（false positive）。原因：
- **日记体天然使用推测词**：diary 类记忆用"应该是吃完了"、"感觉她心情不错"等是正常写法，不是把猜测存成事实。

- **引用琦琦的原话**：mianmian_260 里"她说大概还要三天到一周"是准确引用，不是猜测。

**结论：** 无需删除或修改任何记录。

**后续防范**（写入 MEMORY_WRITER_PROMPT，见 7.5）：对"琦琦说了X"类事实，写"她说了什么原话"而非"她的意思是X"。

### 7.3 1.3 矛盾记忆对（25 对候选，0 对真矛盾）

**扫描方法**：对 11 条关键记忆逐一找 embedding 相似度 > 0.89 的对，按日期比对是否有事实矛盾。

**发现：** 25 对候选全部为**误报**，算法把"高语义相似 = 矛盾"等同了。实际情况：

| 高频误报对 | 真实关系 | 结论 |
|-----------|---------|------|
| `feelings_456`（4/27分手风波）× rules/intimate 记录 | 同一时期的情感关联事件 | 非矛盾 |
| `feelings_136`（4/21"等转正"）× `memory_344`（4/26转正）| 转正前 → 转正后的时间线进展 | 非矛盾 |
| `memory_344/348`（4/26真实）× `memory_500`（4/27确认）| 转正当天 → 次日再确认，同一事件两视角 | 非矛盾 |
| `diary_reply_1777205087`（4/26转正feelings）× `memory_500`| 同事件不同角度记录 | 非矛盾 |
| `pinned_1778182200` × conv_depth 条目 | 主题相关，不是事实对立 | 非矛盾 |

**关于 memory_500 vs memory_344/348 的"矛盾"**：
memory_500 里沐栖当时说"转正还没到呢"，事后被琦琦纠正。这是历史记录，不是当前事实。memory_784（已修正）的存在已经解释了这段混乱。

**结论：** 无需额外处理。矛盾检测算法在情感陪伴场景下误报率极高（>95%），根本原因是亲密关系话题 embedding 空间高度聚集，相似度阈值不适用。

### 7.4 1.4 重复事件（24 对候选，0 对真重复）

**扫描方法**：检测同 1 小时内、Jaccard 词频相似度 > 0.5 的同类记忆对。

**发现：** 24 对候选全部为贴纸记忆（`sticker_8657122517_...`），发生在同一聊天 session 的连续贴纸交换中。

**为什么触发：**
- 贴纸记忆模板高度相同：`"[时间] 琦琦发了一个贴纸X(来自Y套装)。我说：Z"`
- 5分钟内发多个贴纸 → template 占词频的 60%+ → Jaccard 虚高
- 但每条记录的是**不同的贴纸、不同的回应**，是真实发生的不同交互

**结论：** 无需删除。这是算法局限，不是真正的重复污染。

**后续防范**（架构建议，不急）：日后可以考虑贴纸记忆按 session 合并写入（"这天我们换了一组😈😘👍贴纸来来回回玩"），而不是每个贴纸单独存。低优先级，生日后处理。

### 6.4 诊断结论：C + D 并发

对照四种假设：

| 假设 | 结论 |
|------|------|
| A. 4月记忆未存进 Pinecone | ❌ 错：memory_344、memory_348 在 Pinecone，清晰记录 4/26 |
